并发登录下session的处理

在进行性能测试的时候并发请求登录接口时候出现失败的情况，对排查记录做一下总结。

并发登录下session的处理

问题

在进行性能测试的时候并发请求登录接口，除登录接口之外的接口保护策略是使用过滤器从cookie获取用户信息并和服务端进行后续的数据交互。

设置一个线程组，线程主要有两个接口

  1. 登录接口，登录成功之后系统会返回一个set-cookie的响应头
  2. 调用被保护的接口，接口在请求时会携带cookie来让服务器鉴别当前请求是否合法

因此定位到是session在存储的时候出了问题。

解决

检查session存储的代码，发现登录接口使用的session是一个共享的变量，因为 spring 的每个 controller 默认都是单例的，这个session 会被其他线程给共享，在多线程的情况下，极容易出现线程不安全的问题。

现有的session处理方式如下：

 1private HttpSession session;
 2private HttpServletRequest request;
 3
 4@ModelAttribute
 5public void init(HttpSession httpSession, HttpServletRequest request) {
 6	this.session = httpSession;
 7	this.request = request;
 8}
 9//登录controller
10 @PostMapping(value = "public/client-account/v1/login-dev")
11@ApiOperation(value = "用户登录校验(明文)")
12public ResponseData loginDev(@RequestBody OmsLoginDTO loginDTO) throws Exception {
13  OmsUserVO omsUserVO = this.userFacade.checkLogin(loginDTO, false);
14  this.userFacade.log(omsUserVO, request);
15  sessionCache(omsUserVO, session);
16  return ResponseUtil.success(omsUserVO);
17}
18/**
19     * 缓存用户session
20     *
21     * @param omsUserVO
22     * @throws Exception
23     */
24private void sessionCache(OmsUserVO omsUserVO) throws Exception {
25  if (repeatLogin.equals("true")) {
26    httpSession.setAttribute(FindByIndexNameSessionRepository.PRINCIPAL_NAME_INDEX_NAME, omsUserVO.getLoginName());
27  }
28  httpSession.setAttribute("user", JSONObject.toJSONString(omsUserVO));
29  if (null != omsUserVO.getRoles() && omsUserVO.getRoles().size() > 0) {
30    List<String> roleNameList = omsUserVO.getRoles().stream()
31      .map(OmsRoleVO::getRoleName)
32      .collect(Collectors.toList());
33    session.setAttribute("role", JSONObject.toJSONString(roleNameList));
34  }
35}

解决方式有三种：

首先介绍最简单的一种，

使用@Autowired注解

使用@Autowired代替@ModelAttribute

 2private HttpServletRequest request;
 3@Autowired
 4private HttpSession session;
 5

 7//public void init(HttpSession httpSession, HttpServletRequest request) {

对上述代码进行调试

加上了@Autowired注解之后进行debug发现，request并不是原始的HttpServletRequest对象，而是HttpServletRequest的一个代理类。找到代理类的实现如下

实际上Autowire进来的并不是原始的HttpServletRequest对象，而是HttpServletRequest的一个代理类。实际上它会通过

((ServletRequestAttributes) RequestContextHolder.getRequestAttributes()).getRequest()

到这一步骤已经很明显了，那如何看出是线程安全的，继续往下看。

进一步去看RequestContextHolder发现RequestContextHolder是通过ThreadLocal来实现的，可以保证每个线程获取得到的Request对象一定是当前请求的Request对象，从而保证线程安全。

一种是在登录相关接口将session作为参数传递到方法中使用

在请求和存储的时候加上单独的HttpSession session

 1@PostMapping(value = "public/client-account/v1/login-dev")
 2@ApiOperation(value = "用户登录校验(明文)")

 8}

另外一种是将session使用ThreadLocal来处理

重点在于session的初始化设置

 1private static final ThreadLocal<HttpSession> session = new ThreadLocal<>();
 2@ModelAttribute
 3public void init(HttpSession httpSession, HttpServletRequest request) {
 4  ClientAccountController.session.set(httpSession);
 5  this.request = request;
 6}
 7//在涉及到session的获取时候需要使用session.get()
 8/**
 9     * 缓存用户session
10     * @param omsUserVO
11     * @throws Exception
12     */
13private void sessionCache(OmsUserVO omsUserVO) throws Exception {
14  if (repeatLogin.equals("true")) {
15    session.get().setAttribute(FindByIndexNameSessionRepository.PRINCIPAL_NAME_INDEX_NAME, omsUserVO.getLoginName());
16  }
17  session.get().setAttribute("user", JSONObject.toJSONString(omsUserVO));
18  if (null != omsUserVO.getRoles() && omsUserVO.getRoles().size() > 0) {
19    List<String> roleNameList = omsUserVO.getRoles().stream()
20      .map(OmsRoleVO::getRoleName)
21      .collect(Collectors.toList());
22    session.get().setAttribute("role", JSONObject.toJSONString(roleNameList));
23  }
24}

使用上述三种方式任意方式之后，最终请求正常

继续调整请求线程为10，问题没有出现。

总结

1、在并发的情况下要注意考虑共享变量的问题

2、request、Response、session在单例controller是不安全的，采用ThreadLocal可以解决该问题。推荐直接使用spring帮助我们处理好的来使用。

3、建议修改现有的baseController。
